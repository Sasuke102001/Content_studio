# PolyNovea Content Desktop App: Current Engine Behavior

This document audits the legacy, python-script-based generation engine for the PolyNovea Carousel Generator. It details the existing workflows, APIs, rendering pipelines, and file dependencies before starting the modular refactoring.

---

## 1. Generation Pipeline & Logic Flow

The engine operates on a single user input prompt generated by loading and concatenating multiple plain text and Markdown files. 

```
[Prompt Parts] ──> Concatenate ──> NVIDIA NIM API (Kimi 2.6) ──> HTML Output ──> Playwright (Chromium) ──> PNG Slides ──> img2pdf (LinkedIn only) ──> PDF
```

### Step 1: Prompt Concatenation
The engine loads the following text inputs and formats them into a single payload:
1. **Brand Visual System**: `context/polynovea_brand_visual_and_content_system_v_1.md`
2. **Platform & Voice Contexts**:
   - For Personal mode: `context/roys-personal-voice.md` and `context/carousel-content-context.md`
   - For Biz modes: No extra context.
3. **Carousel Content Request**: `carousel_request.txt` from the specific post folder.
4. **HTML Generation Rules / Design System**: `html_request.txt` from the specific post folder.
5. **Critical Output Rules**: Slide dimensions, slide count parameters, HTML class structures, and completion instructions.

### Step 2: NVIDIA NIM API Request
The combined prompt is dispatched to Moonshot Kimi via NVIDIA's NIM API:
* **Endpoint**: `https://integrate.api.nvidia.com/v1/chat/completions`
* **Model**: `moonshotai/kimi-k2.6` (sometimes referenced as Kimi 2)
* **Temperature**: 0.6
* **Max Tokens**: 8,000 (limits can cause truncated output if prompts are too large)
* **Streaming**: Enabled. Tokens are streamed and printed to the terminal in real-time, then aggregated.

### Step 3: HTML Post-Processing & Validation
The engine extracts HTML from the response, removing any markdown code block fences (e.g., ` ```html ` and ` ``` `). It then runs basic safety validation checks:
* Ensures opening and closing structural tags (`<body`, `</body>`, `</html>`) are present.
* Fails early if the tags are missing, alerting the developer to LLM truncation.
* Saves the sanitized HTML string to `outputs/[section]/post_[id]/[platform]/carousel.html`.

### Step 4: PNG Export (Screenshotting)
The engine spawns a headless Chromium instance using the Playwright sync API:
1. Viewport is configured to target slide dimensions (e.g., 540x540px for LinkedIn, 420x525px for Instagram).
2. The browser `device_scale_factor` is set to `1080 / viewport_width` (e.g., `2.0` or `2.571`) to scale up image resolution to high-definition (1080px wide) without reflowing font and layout structures.
3. The page loads the local file URL of the generated `carousel.html`.
4. The runner waits **3000ms** to allow web fonts (like Clash Display) to download from the Fontshare CDN.
5. It queries all elements matching `.slide`.
6. Each matching element is captured as a PNG screenshot and saved sequentially in `/slides/slide-XX.png`.

### Step 5: PDF Assembly (LinkedIn-only)
LinkedIn carousels must be uploaded as a single multi-page PDF. For LinkedIn posts, the engine uses the `img2pdf` library:
- Scans and sorts all screenshots in the `/slides/` folder.
- Compiles the sorted PNGs into a single margin-free PDF.
- The output PDF size matches the slide images exactly, ensuring no white padding or margin blocks are added.

## 2. Environment & Configuration Dependencies

- **API Keys**: Loaded from the root `.env` file via `python-dotenv`:
  ```bash
  NVIDIA_API_KEY="nvapi-..."
  ```
- **Fonts & Assets**: The HTML output references Fontshare CDN for fonts, but local assets like fonts can be housed in `assets/fonts/`.
- **System Packages**: Node/Electron is not used in the legacy engine; all orchestration runs through Python 3 standard libraries, `requests`, `playwright`, and `img2pdf`.
